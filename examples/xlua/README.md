# NativeRelay × xLua integration example

**English** · [简体中文](README.zh-CN.md)

Reference boilerplate for driving the [NativeRelay](https://github.com/forestlii/NativeRelay)
bridge entirely from **Lua** (via [xLua](https://github.com/Tencent/xLua)) — every business
call, plus success / failure / error-code handling, lives in Lua.

> **Status — reference sample, not compiled here.** This repo has no xLua installed, so these
> files are not built/verified in CI. The C# signatures match NativeRelay's frozen public
> contract (v0.2.0). Build + on-device verify in your own xLua project.

## Files

| File | Role |
|---|---|
| [`NativeRelayXLuaConfig.cs`](NativeRelayXLuaConfig.cs) | xLua generation config — **required**, see the trap below |
| [`native_relay.lua`](native_relay.lua) | Thin Lua wrapper over the C# bridge; funnels all code handling |
| [`business_example.lua`](business_example.lua) | Pure-Lua usage: device info / permission / location |

The command codes (`Cmd.GetDeviceInfo`, …) come from `relay_command.lua`, generated by
[`tools/codegen`](../../tools/codegen) from `commands.json` — the single source of truth across
C# / Java / Obj-C / Lua. Don't hand-copy the numbers.

## Setup (3 steps)

1. Install xLua in your Unity project.
2. Drop `NativeRelayXLuaConfig.cs` under `Assets/`, then run **XLua → Generate Code**.
3. Load `native_relay.lua` + `relay_command.lua` into your Lua require path; write business
   logic like `business_example.lua`.

## The one trap that bites everyone

On **IL2CPP / AOT** (iOS, Android-IL2CPP), passing a Lua function where a C# `Action<int,string>`
is expected **throws at runtime** unless that delegate type is registered in `[CSharpCallLua]`
and code is regenerated. It works in the Editor (Mono, reflection), so it slips through until you
build to device. `NativeRelayXLuaConfig.cs` registers it — don't skip the code-gen step.

## Things worth knowing

- **Threading:** `onResult` is dispatched on the **main thread** by the bridge, so the Lua VM
  (not thread-safe) is never touched concurrently — you don't need to marshal threads yourself.
- **Allocation:** each `request` allocates a Lua closure + a C# delegate wrapper. Native
  capabilities (permission/photo/location) are low-frequency, so this is fine — not a hot path.
- **Lifecycle:** in-flight requests are held until the result or timeout returns; disposing the
  bridge mid-flight delivers the `RelayCode.Disposed` code to each pending callback exactly once.
